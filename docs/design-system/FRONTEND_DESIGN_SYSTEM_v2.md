# Frontend Design System v2.0
## «Интуитивная плотность» (IA-first Dense UX)

Главная цель:

> **Минимум места. Максимум понятной информации. Минимум мыслительного усилия.**

Важно: **плотно ≠ тесно**. Плотность должна быть не визуальной, а смысловой.

---

## 0. Главная формула системы

```text
Intuitivnost' =
  ponyatnaya struktura
+ znakomye patterny
+ vizual'naya ierarkhiya
+ korotkiy tekst
+ mgnovennaya obratnaya svyaz'
+ dostupnost'

Khoroshaya plotnost' = mnogo smysla / malo shuma
Plokhaya plotnost' = mnogo elementov / malo ponimaniya
```

## 1. Приоритеты интерфейса (P0–P3)

### P0 — критическое
То, без чего пользователь не понимает экран.
- где я нахожусь
- что это за страница
- главное действие
- ключевая информация
- ошибка, если она есть

**P0 всегда видно сразу.**

### P1 — частое
То, что используется часто.
- фильтры, сортировка, поиск
- основные карточки, список объектов

**P1 видно без глубокого поиска.**

### P2 — дополнительное
То, что нужно иногда.
- расширенные настройки, вторичные действия, справка

**P2 можно прятать во вкладки, аккордеоны, drawer.**

### P3 — редкое
То, что почти никогда не нужно.
- опасные действия, архив, технические настройки, админские функции

**P3 убирается глубже, но не исчезает полностью.**

## 2. Дизайн-токены (default) — БЕЗРАМОЧНЫЙ (borderless) стиль

> **Принцип: мы не используем видимые границы (бордеры) для разделения контента. Группировка происходит через whitespace, background tint, и typography hierarchy.**

### 2.1. Философия безрамочного UI

```text
Плохо: ┌─────────┐  ┌─────────┐  ┌─────────┐
        │ блок 1  │  │ блок 2  │  │ блок 3  │  ← видимые прямоугольники
        └─────────┘  └─────────┘  └─────────┘

Хорошо: Заголовок блока 1
         Контент контент контент

        Заголовок блока 2
         Контент контент контент
        
        ← разделение через whitespace + заголовки
```

**Правила безрамочного layout:**
1. **Нет видимых границ** для карточек, панелей, ячеек таблиц.
2. **Разделение через whitespace** — отступы между группами 24–48px, внутри группы 8–16px.
3. **Разделение через фон** — мягкие background tints (neutral-50 vs white) вместо border.
4. **Разделение через типографику** — font-weight, size, color hierarchy.
5. **Таблицы без сетки** — только горизонтальные разделители (1px, 10% opacity), без вертикальных.
6. **Inputs без box-shadow** — только bottom-border или tint, без "boxy" обрамления.
7. **Кнопки без border** — background tint для secondary, solid для primary.
8. **Модальные окна без жёсткой рамки** — мягкий shadow, без border.

### 2.2. Единицы
```text
1 rem = 16px
Bazovyy shag = 4px
Osnovnaya setka = 8px
```

### 2.2. Spacing scale

| Токен | px | Где использовать |
|---|---|---|
| `space-0` | 0 | Без отступа |
| `space-0-5` | 2 | Микро-коррекция |
| `space-1` | 4 | Иконка внутри элемента |
| `space-2` | 8 | Иконка + текст |
| `space-3` | 12 | Плотные строки |
| `space-4` | 16 | Базовый padding карточки |
| `space-5` | 20 | Средний внутренний padding |
| `space-6` | 24 | Группы, desktop padding |
| `space-8` | 32 | Крупные блоки |
| `space-10` | 40 | Большой вертикальный отступ |
| `space-12` | 48 | Разделы страницы |
| `space-16` | 64 | Большие секции |
| `space-20` | 80 | Очень крупные разделы |
| `space-24` | 96 | Landing spacing |
| `space-32` | 128 | Редко, промо-блоки |

**Правила отступов:**
1. Между иконкой и текстом: **8px**.
2. Между label и input: **6–8px**.
3. Padding обычной кнопки: **12px 16px**.
4. Padding карточки: **16px** compact, **24px** default.
5. Между карточками: **16px** compact, **24px** default.
6. Между смысловыми секциями: **32–48px**.
7. Между заголовком и текстом: **8–12px**.
8. Внутри группы расстояние всегда меньше, чем между группами.
9. Не использовать случайные значения (13px, 17px, 29px).

## 4. Типографика

### Шрифт по умолчанию
```text
Inter
```
Fallback: `ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif`

### Типографическая шкала
| Название | Desktop | Mobile | Weight | Где использовать |
|---|---|---|---|---|
| Display | 56/64 | 40/48 | 700 | Главные промо-заголовки |
| H1 | 40/48 | 32/40 | 700 | Заголовок страницы |
| H2 | 32/40 | 28/36 | 700 | Большие секции |
| H3 | 24/32 | 22/30 | 600 | Подсекции |
| H4 | 20/28 | 20/28 | 600 | Заголовки карточек |
| Body L | 18/28 | 18/28 | 400 | Длинный текст |
| Body | 16/24 | 16/24 | 400 | Основной текст |
| UI | 14/20 | 14/20 | 400/500 | Интерфейс, таблицы |
| Caption | 12/16 | 12/16 | 400/500 | Подписи, метаданные |
| Tiny | 11/14 | — | 500 | Только бейджи, редко |

### Правила текста
1. Основной читаемый текст: **16px минимум**.
2. UI-текст в таблицах: **14px**, но не меньше.
3. Caption: **12px** только для вторичной информации.
4. Длина строки: **50–75 символов**.
5. Для длинного текста max-width: **680–760px**.
6. Не использовать `font-weight: 300` для интерфейса.
7. Основные веса: **400, 500, 600, 700**.
8. Заголовки должны быть короче 60 символов.
9. Кнопки — 1–3 слова.
10. Не использовать капслок для длинных фраз.

3. Зелёный только для: успеха, подтверждения, положительного статуса.
4. Primary color нельзя использовать для всего подряд. Он означает: «сюда нужно нажать».
5. Фон: page background `neutral-50`, cards `white`, border `neutral-200`.
6. Контраст: обычный текст минимум **4.5:1**, крупный текст **3:1**, иконки/границы **3:1**.

## 6. Радиусы, тени, бордеры

### Border radius
| Токен | px | Где использовать |
|---|---|---|
| `radius-none` | 0 | Таблицы, строгий UI |
| `radius-sm` | 4 | Маленькие элементы |
| `radius-md` | 6 | Inputs |
| `radius-lg` | 8 | Buttons, cards |
| `radius-xl` | 12 | Modals, large cards |
| `radius-2xl` | 16 | Promo blocks |
| `radius-full` | 999 | Pills, avatars |

Default: buttons 8px, inputs 6–8px, cards 12px, modals 16px, badges 999px.

### Borders — минимально, прозрачно, или не использовать
```text
Default border: none (или 1px solid rgba(226, 232, 240, 0.25) — 25% opacity)
Strong border: none (или 1px solid rgba(203, 213, 225, 0.40) — 40% opacity)
Focus ring: 2px solid #2563EB (outline, не border)
Error indicator: background tint + text color, не border
```

**Правила borderless:**
1. **Нет видимых border для карточек, панелей, ячеек.**
2. **Inputs** — bottom-border или background tint, не box outline.
3. **Разделители** — whitespace (16–48px) вместо `<hr>`.
4. Если нужен разделитель — очень мягкий (1px, 10–15% opacity).
5. **Active/selected state** — background tint, не border-color change.

### Shadows — только для floating элементов
| Токен | CSS | Где использовать |
|---|---|---|
| `shadow-xs` | `0 1px 2px rgba(15, 23, 42, 0.05)` | Subtle elevation |
| `shadow-sm` | `0 1px 3px rgba(15, 23, 42, 0.10)` | Dropdown, popover |
| `shadow-md` | `0 4px 12px rgba(15, 23, 42, 0.12)` | Modal, dialog |
| `shadow-lg` | `0 12px 32px rgba(15, 23, 42, 0.16)` | Full-screen overlay |

**Правила:**
1. Тени только для **floating элементов** (dropdown, modal, popover, toast).
2. Карточки, панели, таблицы — **не используют shadow** (это создаёт "поднятые коробки" эффект).
3. Разделение через **whitespace + background tint**, не border + shadow.

## 7. Motion / анимации

### Длительности
| Тип | Время |
|---|---|
| Hover/focus | 100–150ms |
| Small UI transition | 150–200ms |
| Dropdown/modal | 200–250ms |
| Page transition | 250–350ms |

### Easing
Default: `cubic-bezier(0.2, 0, 0, 1)`  
Для выхода: `cubic-bezier(0.4, 0, 1, 1)`

Правила:
1. Анимация помогает понять изменение, а не украшает.
2. Не анимировать всё подряд.
3. Уважать `prefers-reduced-motion`.
4. Если действие мгновенное — не делать длинную анимацию.
5. Если ожидание больше 300–500ms — показать feedback.

## 8. Компоненты

### 8.1. Кнопки

#### Размеры
| Размер | Height | Padding | Font |
|---|---|---|---|
| Small | 32px | 8px 12px | 14px |
| Medium | 40px | 10px 16px | 14px |
| Large | 48px | 12px 20px | 16px |

Default: height 40px, radius 8px, font-weight 500/600, gap icon-text 8px.

#### Типы
1. `Primary` — главное действие.
2. `Secondary` — вторичное действие.
3. `Ghost` — тихое действие.
4. `Danger` — удаление/опасность.
5. `Icon button` — только иконка.
6. `Link button` — действие выглядит как ссылка.

#### Правила
1. На экране максимум **1 primary action**.
2. Primary button ставится справа в форме/modal или сверху справа в page header.
3. Danger button не должен быть рядом с primary без визуального разделения.
4. Disabled button должен иметь объяснение, почему он disabled.
5. Icon-only button всегда имеет tooltip/aria-label.
6. Минимальная click/tap зона: **44×44px**. Даже если визуально 32px, hit area больше.

### 8.2. Inputs

#### Размеры
| Размер | Height | Padding |
|---|---|---|
| Compact | 36px | 8px 12px |
| Default | 40px | 10px 12px |
| Large | 48px | 12px 14px |

Default: height 40px, radius 8px, border `#CBD5E1`, focus `#2563EB`, error `#DC2626`.

#### Поле должно иметь
- label (сверху, не внутри поля)
- placeholder только как пример
- help text, если поле сложное
- error message (что не так + как исправить)
- success state, если нужно
- disabled/read-only state

#### Ошибка поля
Плохо: `Ошибка`.  
Хорошо: `Введите email в формате name@example.com`.

Формула: **Что не так + как исправить.**

### 8.3. Forms
Правила:
1. Label сверху, не внутри поля.
2. Один главный action в конце формы.
3. Ошибки показывать рядом с полем.
4. Валидация: во время ввода — осторожно; после blur — хорошо; после submit — обязательно.
5. Не делать форму в 3 колонки.
6. На mobile всегда 1 колонка.
7. На desktop можно 2 колонки, если поля короткие.
8. Обязательные поля отмечать явно.
9. Не прятать важные поля в placeholder.
10. После успешного действия показать результат.

### 8.4. Tables — Borderless (безрамочные)
Для информационно-плотного UI таблицы очень важны. **Без видимых границ ячеек.**

| Элемент | Размер |
|---|---|
| Header row | 40px |
| Default row | 44px |
| Compact row | 36px |
| Cell padding | 8px 12px |
| Font | 14px |
| Row separator | 1px `neutral-200` at 15% opacity (только горизонтальный) |
| **Vertical borders** | **none** |

Правила:
1. **Нет вертикальных границ.** Только мягкие горизонтальные разделители.
2. **Альтернативные строки** — мягкий background tint (neutral-50/100) для читаемости.
3. Header — отделяется от body **whitespace + background tint**, не border.
4. Текст слева.
5. Числа справа.
6. Даты справа или слева, но единообразно.
7. Статусы — badge.
8. Действия — справа.
9. Header sticky, если таблица длинная.
10. Первая важная колонка может быть sticky.
11. Сортировка видна и понятна.
12. Фильтры над таблицей (не внутри, не в рамках).
13. Empty state вместо пустого белого пространства.
14. Pagination или infinite scroll — не смешивать без причины.
15. В таблице не делать много ярких цветов.
16. Long text truncate + tooltip/details.
17. Для массовых действий — checkbox selection.
18. Compact mode только на desktop.

### 8.5. Cards — Borderless
Default card (безрамочная):
```text
background: white (или subtle tint для группировки)
border: none
radius: 12px
padding: 20px или 24px
gap: 12px или 16px
shadow: none (кроме floating — dropdown, modal, popover)
```

**Правила borderless cards:**
1. **Нет border.** Разделение через background tint + whitespace.
2. В карточке один главный смысл.
3. Заголовок сверху (font-weight 600 + size H3/H4).
4. Действия — справа сверху или внизу (secondary buttons, не primary).
5. Не помещать в карточку всё подряд.
6. Карточки одного типа — одинаковой структуры.
7. Если карточек много и однотипных — возможно, нужна таблица.
8. **Карточки на одном фоне** — используй background tint (neutral-100) для card, а не border.
9. **Padded regions** — вместо "панелей в панелях", используй whitespace.

### 8.6. Navigation

#### Header
```text
Desktop height: 64px
Mobile height: 56px
```
Содержит: logo, primary navigation, search (если большой сайт), user/account actions, mobile menu button.

#### Sidebar — Borderless
```text
width: 280px
collapsed: 72px
nav item height: 40px
icon: 20px
gap icon-text: 12px
background: var(--surface) или transparent (не box)
border-right: none (или 1px at 10% opacity)
```

Правила:
1. **Нет видимой рамки.** Sidebar — это area, не box.
2. Активный раздел подсвечен — **background tint** (primary-50), не border.
3. Hover — **background tint**, не border highlight.
4. Иконка без текста — только в collapsed mode.
5. Вложенность не глубже 2 уровней в sidebar.
6. Если уровней больше — нужны отдельные страницы-разделы.
7. Sidebar не должен конкурировать с главным контентом.
8. **Разделительные группы** — whitespace (16px), не `<hr>` или border.

### 8.7. Breadcrumbs
Использовать, если глубина больше 2 уровней.
```text
Главная › Раздел › Подраздел › Текущая страница
```
Правила: последний элемент не ссылка; названия совпадают с заголовками страниц; на mobile можно сокращать середину.

### 8.8. Modals — Borderless, shadow-based
| Тип | Width |
|---|---|
| Small | 400px |
| Medium | 560px |
| Large | 720px |
| XL | 960px |

**Borderless modal styling:**
```text
border: none
background: var(--surface)
shadow: shadow-lg (только floating — допустим)
radius: radius-2xl (16px) для мягкости
padding: 24px или 32px
```

Правила:
1. Modal — для короткой задачи.
2. Длинные формы — отдельная страница или drawer.
3. Заголовок обязателен (H3, font-weight 600).
4. Primary action справа снизу.
5. Cancel — ghost button (без border, тихий), слева от primary.
6. Escape закрывает, если нет потери данных.
7. При несохранённых данных — предупреждать (inline, не alert box).
8. Не открывать modal поверх modal.
9. **Overlay backdrop** — мягкий, затемнённый (rgba(0,0,0,0.3)), не резкий.
10. **Нет border внутри modal** — разделители через whitespace + typography.

### 8.9. Toast / уведомления
Default:
```text
width: до 420px
position: top-right desktop, bottom mobile
duration: 4–6 seconds
```

Правила:
1. Toast не должен содержать критически важную информацию, которая исчезнет навсегда.
2. Success toast — короткий.
3. Error toast — должен иметь действие или путь исправления.
4. Важное действие — inline alert, не toast.
5. Не показывать больше 3 toast одновременно.


### Правила дерева сайта
1. Главные разделы: **5–7 пунктов**.
2. Глубина для 80% задач: максимум **3 клика**.
3. Каждый раздел отвечает на один пользовательский вопрос.
4. Названия разделов — словами пользователя, не внутренним жаргоном.
5. Не смешивать разные принципы группировки (продукты, аудитории, отделы).
6. В каждом разделе должна быть обзорная страница.
7. На глубоких страницах — breadcrumbs.
8. Поиск обязателен, если страниц/объектов много.
9. Не должно быть тупиковых страниц.
10. В конце страницы всегда есть следующий логичный шаг.

## 10. Шаблон каждой страницы

Каждая страница должна иметь:
```text
1. Page title (H1)
2. Kratkoe opisanie / kontekst
3. Primary action
4. Osnovnoy kontent
5. Secondary actions
6. Sostoyaniya: loading, empty, error, success
7. Navigatsionnyy put' nazad/vverkh
```

### Page header
Desktop: title слева, description под title, primary action справа, secondary actions рядом или в menu.  
Mobile: title сверху, action ниже или sticky bottom.

```text
Disabled
Loading
Empty
Error
Success
Read-only
```

Если состояние не описано — программист будет придумывать сам. Это плохо.

## 12. Алгоритм проектирования страницы

### Шаг 1. Определи топ-задачи (5–10)
```text
1. Nayt' zakaz
2. Pomenyat' status zakaza
3. Dobavit' novogo klienta
4. Prosmotret' karta kuryerov
5. Proverit' finansy
```

### Шаг 2. Построй дерево сайта
Сначала sitemap, потом дизайн.

### Шаг 3. Для каждой страницы определи P0/P1/P2/P3
Пример: Страница заказа
```text
P0: nomer zakaza, status, klient, glavnoe deystvie
P1: adress dostavki, kuryer, vremya, summa
P2: istoriya izmeneniy, audit, kommentarii
P3: arkhiv, tekhn. dannye, idempotentnost'
```

### Шаг 4. Сделай wireframe без цвета
Если экран непонятен без цвета и картинок — структура плохая.

### Шаг 5. Примени токены
Только после структуры: цвета, шрифты, отступы, карточки, кнопки.

### Шаг 6. Проверь состояния
```text
Loading? Empty? Error? Success? Mobile? Long text? No data? Too much data?
```

### Шаг 7. Протестируй
Дай человеку задачу: «Найди заказ и измени статус». Не объясняй интерфейс. Если он спрашивает, куда нажать — проблема в дизайне.

## 13. Definition of Done для страницы

Страница готова, если:
- [ ] понятно, где пользователь находится
- [ ] есть H1
- [ ] есть главное действие
- [ ] не больше одного primary action
- [ ] структура совпадает с sitemap
- [ ] все отступы из spacing scale
- [ ] все цвета из tokens
- [ ] текст читаемый
- [ ] есть mobile версия
- [ ] есть loading state
- [ ] есть empty state
- [ ] есть error state
- [ ] работает keyboard navigation
- [ ] виден focus
- [ ] контраст соответствует WCAG AA
- [ ] нет тупика в пользовательском пути

## 15. 100 senior-законов UI (сжатая сводка)

### Физика экрана и восприятия (1–10)
1. Закон CSS-пикселя — проектируй в CSS px, а не физических.
2. Закон viewport'ов — проверяй на 320, 390, 768, 1366, 1440, 1920.
3. Закон визуального угла — чем дальше экран, тем крупнее элементы.
4. Закон контраста — минимум 4.5:1 для текста.
5. Закон тонких линий — не используй светлый текст тонким весом (300 + серый = плохо).
6. Закон антишума — убирай всё, что не помогает задаче.
7. Закон стабильной сетки — глаз быстрее считывает повторяющиеся линии.
8. Закон размера цели — минимальная зона клика 44×44px.
9. Закон расстояния до цели — частые действия ближе к точке решения.
10. Закон края экрана — края и углы — опорные зоны.

### Биология зрения и моторики (11–20)
11. Закон фовеального зрения — детали видны только в центральной зоне.
12. Закон периферии — периферия замечает движение и контраст, но плохо читает текст.
13. Закон цветовых ограничений — не все различают красный/зелёный (цвет + иконка + текст).
14. Закон утомления глаз — длинный текст на ярком белом без воздуха утомляет.
15. Закон мелкой моторики — на touch мелкие элементы вызывают ошибки (mobile 44–48px).
16. Закон большого пальца — важные действия ближе к нижней части экрана.
17. Закон биологического ритма внимания — нужны заголовки, группы, паузы.
18. Закон усталости от выбора — показывай сначала частые, редкие в «More».
19. Закон мгновенной обратной связи — мозг ждёт реакции сразу.
20. Закон избегания боли — ошибка должна помогать, а не пугать.

### Психология и когнитивная нагрузка (21–35)
21. Закон «не заставляй думать» — если спрашивают «куда нажать?» — UI плох.
22. Закон узнавания вместо памяти — показывай варианты, не заставляй вспоминать.
23. Закон Хика — чем больше вариантов, тем дольше решение. Группируй.
24. Закон 4±1 чанков — в одном блоке 3–5 ключевых элементов.
25. Закон когнитивной нагрузки — не перегружай одновременно чтением, поиском, сравнением.
26. Закон одного главного действия — на экране один primary action.
27. Закон последовательного раскрытия — сначала суть, потом детали.
28. Закон предсказуемости — похожие элементы ведут себя одинаково.
29. Закон привычных паттернов — не изобретай select/checkbox/tabs без нужды.
30. Закон ментальной модели — структура совпадает с мышлением пользователя.
31. Закон эффекта первого экрана — объясняет: где я, что есть, что делать.
32. Закон серийной позиции — важные пункты в начало и конец списка.
33. Закон дефолтов — хорошие значения по умолчанию уменьшают решения.
34. Закон потери — пользователи боятся потерять больше, чем хотят получить.
35. Закон доверия — непоследовательный UI снижает доверие.

### Гештальт и визуальная организация (36–45)
36. Закон близости — связанные элементы ближе друг к другу.
37. Закон подобия — одинаковые элементы выглядят одинаково.
38. Закон общей области — элементы внутри одной карточки = группа.
39. Закон непрерывности — глаз идёт по линиям. Используй выравнивание.
40. Закон фигуры и фона — главный объект ясно отделяется.
41. Закон иерархии — размер, вес, цвет, позиция показывают важность.
42. Закон визуального веса — крупный, тёмный, насыщенный = важнее.
43. Закон пустого пространства — whitespace — инструмент группировки.
44. Закон ритма — повторяющиеся размеры и отступы = порядок.
45. Закон якорей — header, sidebar, breadcrumbs, title — стабильные точки.

### Информационная архитектура (46–55)
46. Закон карты сайта — нельзя проектировать экраны без дерева сайта.
47. Закон 5–7 разделов — главное меню в диапазоне 5–7 пунктов.
48. Закон 3 кликов — ключевые задачи за 1–3 шага.
49. Закон «ты здесь» — пользователь всегда понимает своё положение.
50. Закон хлебных крошек — если глубина > 2 уровней — нужны breadcrumbs.
51. Закон названий пользователя — разделы на языке пользователя, не компании.
52. Закон отсутствия тупиков — каждая страница предлагает следующий шаг.
53. Закон поиска как страховки — если не понял структуру, поиск спасёт.
54. Закон одинаковых названий — один объект = одинаковое название везде.
55. Закон информационного запаха — ссылка ясно обещает, что будет после клика.

### Контент и язык (56–65)
56. Закон коротких фраз — чем короче, тем быстрее сканируется.
57. Закон глагола в кнопке — отвечает на вопрос «что сделать?».
58. Закон конкретики — «Ошибка сохранения» хуже, чем «Не удалось сохранить: нет подключения».
59. Закон полезного заголовка — объясняет смысл блока, а не декоративный.
60. Закон сканирования — люди сначала сканируют, потом читают. Списки, подзаголовки.
61. Закон одного термина — не называй один объект «заказ», «заявка», «запрос».
62. Закон без канцелярита — «Осуществить отправку данных» → «Отправить данные».
63. Закон чисел — `1 200 000` → `1,2 млн`, если точность не важна.
64. Закон дат — единообразно: `12.06.2026` или `12 июня 2026`, но не смесь.
65. Закон пустого состояния — empty state объясняет, что здесь будет и как начать.

### Цвет (66–70)
66. Закон семантики цвета — цвет должен иметь смысл, а не просто украшать.
67. Закон акцентной экономии — если всё синее — ничего не синее. Акцент только для важного.
68. Закон красного цвета — красный: только ошибка, опасность, удаление.
69. Закон фона — контент лучше читать на спокойном фоне (white, neutral-50, neutral-100).
70. Закон тёмной темы — тёмная тема ≠ инверсия. Не используй чистый `#000`.

### Типографика (71–75)
71. Закон читаемого body — основной текст 16px/24px.
72. Закон плотного UI — для таблиц и control labels допустимо 14px/20px.
73. Закон line-height — для текста 1.4–1.6, для UI 1.2–1.4.
74. Закон длины строки — длинный текст 50–75 символов в строке.
75. Закон табличных цифр — в колонках с числами используй tabular numbers.

### Layout и плотность (76–80)
76. Закон 8px-сетки — основные расстояния кратны 8px, мелкие — 4px.
77. Закон группировки — внутри группы 8–16px, между группами 24–48px.
78. Закон верхнего приоритета — самое важное выше и левее в desktop.
79. Закон адаптации — на mobile не сжимай desktop, а перестраивай layout.
80. Закон прогрессивной плотности — новичку проще, опытному — compact mode.

### Взаимодействие (81–90)
81. Закон состояния системы — после действия пользователь должен видеть результат.
82. Закон загрузки — если загрузка > 300–500ms, показывай skeleton/spinner.
83. Закон undo — лучше дать «Отменить», чем спрашивать подтверждение на каждое действие.
84. Закон опасного действия — удаление требует подтверждения или возможности восстановления.
85. Закон disabled-состояния — если кнопка disabled, объясни почему.
86. Закон keyboard-first — всё должно работать с клавиатуры.
87. Закон focus ring — focus должен быть видимым.
88. Закон hover ≠ доступность — информация не должна быть доступна только при hover.
89. Закон mobile gestures — жесты должны иметь видимую альтернативу.
90. Закон не ломать Back — кнопка браузера «назад» должна вести ожидаемо.

### Доступность и инженерия (91–100)
91. Закон WCAG AA — минимальная цель.
92. Закон семантики HTML — кнопка = `<button>`, ссылка = `<a>`, не div с onClick.
93. Закон alt-текста — смысловые изображения имеют alt, декоративные — пустой alt.
94. Закон масштабирования — интерфейс должен работать при zoom 200%.
95. Закон reduced motion — уважай `prefers-reduced-motion`.
96. Закон скорости — интерфейс должен реагировать до 100ms, если возможно.
97. Закон Core Web Vitals — LCP < 2.5s, CLS < 0.1, INP < 200ms.
98. Закон ошибок разработки — каждый компонент должен иметь loading, error, empty state.
99. Закон документации — если компонент не описан — он будет использован неправильно.
100. Закон тестирования — проверяй интерфейс на реальных людях. Даже 5 пользователей часто находят основные проблемы.

## 16. Главное правило всей системы

Если сомневаешься, что выбрать, используй такой порядок приоритетов:

```text
1. Dostupnost' (accessibility)
2. Ponyatnost'
3. Skorost' vypolneniya zadachi
4. Konsistnost'
5. Informatsionnaya plotnost'
6. Krasota
```

Красота важна, но она не должна ломать понимание.

## 17. Финальная формулировка

> **Информационно-плотный функциональный минимализм, построенный на сильной информационной архитектуре, прогрессивном раскрытии и дизайн-токенах.**

Или короче:
> **IA-first Dense UX Design System.**

Это подход для SaaS, dashboard, admin panel, delivery operations, marketplaces и любых сложных интерфейсов, где важно уместить много информации, но не перегрузить человека.
